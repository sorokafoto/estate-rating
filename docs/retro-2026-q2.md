## Часть I. Контекст и итоги Q2

### Параметры цикла

| Параметр | Значение |
|----------|----------|
| Застройщиков в легенде | **100** |
| План заявок | **2 100** (100 × 21) |
| Окно аналитики | **72 часа** после отправки заявки |
| Период отправки | **2–8 июня 2026** |
| SIM для заявок | **21** уникальный номер |
| Каналы входящих | звонки, SMS, WhatsApp, Telegram, Max |

После последней волны заявок ждали ещё ~3 суток, чтобы перезвоны по последним заявкам попали в окно 72 ч.

### Что получилось хорошо

- **Единый пайплайн** `update-rating`: от `source.xlsx` до публичного `data.json` без утечки PII в браузер.
- **Публичный рейтинг + внутренний дашборд** на одних агрегатах; `company-events/` — детализация для встреч с застройщиком.
- **Разделение метрик** «первый звонок» (`avg_call_response`) и «любой контакт» (`avg_response`).
- **Инфраструктура идентификации**: `phone_registry`, `phones_to_identify.xlsx`, `classify-phones`, `export-phones-identify`.
- **74 застройщика** с кворумом для сравнительного рейтинга при плане 100.

### Что было сложно

- **~45% звонков не идентифицированы** — главный лимит достоверности рейтинга (много спама и неидентифицированных).
- **−120 заявок** от плана (1 980 / 2 100): формы, капча, редиректы, неверный URL.
- **26 застройщиков** с `insufficient_data` — часто молчание или провал идентификации при полных 21 заявке.
- **28% id. звонков — сироты** (не попали в рейтинг при известном застройщике).

### 4. Выходные vs будни (медиана первого звонка)

**Что считаем.** Для каждой отправленной заявки берём **первый идентифицированный звонок застройщика** (`identified_status = developer`, канал `call`) в окне **0…72 ч** после `application_datetime`. Метрика среза — **медиана** этих задержек (минуты до первого звонка). SMS / WhatsApp / Telegram здесь **не участвуют** — только звонок, как в `avg_call_response`.

**Что такое N.** **N** — число заявок в срезе, у которых **такой звонок вообще был**. Это **не** все 1 980 отправленных заявок:

| | Заявок |
|--|--------|
| Отправлено за цикл | **1 980** |
| С первым id. звонком в 72 ч (N по рынку) | **957** (694 будни + 263 выходные) |
| Без id. звонка в 72 ч | **~1 023** (молчание, только мессенджеры, провал идентификации) |

То есть таблица ниже описывает **скорость ответа там, где ответ звонком уже случился**, а не долю молчащих. (Отдельно: доля заявок без касания на выходных и в будни почти одинакова — ~49% vs ~47%.)

**Как режем срезы** (поля `day_of_week` / `time_slot` из `source.xlsx`, MSK):

| Срез | Правило |
|------|---------|
| **Будни** | пн–пт (`monday`…`friday`) |
| **Выходные** | сб–вс (`saturday`, `sunday`) |
| **Утро / день / вечер** | `morning` / `afternoon` / `evening` по слоту отправки заявки |

Распределение **всех** отправленных заявок: **1 434** будни / **546** выходные (27,6% в выходные).

| Срез | N | Медиана, мин | Комментарий |
|------|---|--------------|-------------|
| Будни | 694 | **~162** | базовая линия |
| Выходные | 263 | **~943** | **~5,8×** медленнее будней |
| Утро | 344 | **~88** | быстрее всего |
| День | 303 | **~183** | близко к будням |
| Вечер | 310 | **~811** | близко к выходным по медиане |

**Интерпретация.** На рынке в целом заявки, отправленные в **субботу/воскресенье** или **вечером**, чаще получают первый звонок **на следующий рабочий день**, а не в тот же день. Медиана ~943 мин (~16 ч) на выходных согласуется с «ответ в понедельник утром».

#### Сравнение по застройщикам

Скрипт [`export-retro-metrics.mjs`](../scripts/export-retro-metrics.mjs) сравнивает медиану будни vs выходные **внутри одного застройщика**.

| Параметр | Значение |
|----------|----------|
| В выборку попадает застройщик, если | ≥ **3** заявки с первым id. звонком **и** в будни, **и** в выходные (`MIN_SLICE_N = 3`) |
| Застройщиков сравнимо | **52** из 100 |
| «Хуже на выходных» | медиана выходных **>** медианы будней (`weekend_slower = true`) |
| **Исключения** | медиана выходных **≤** медианы будней — на выходных **не медленнее** (иногда быстрее); топ по наименьшему отношению выходные/будни |

**Топ «хуже на выходных»** (медиана выходных / медиана будней, по убыванию ratio): ФСК, ПИК, Setl Group, КВС, ЛСР — типичный паттерн: будни ~1–2 мин, выходные ~12–16 ч.

**Исключения** (ответ на выходных не хуже будней; в ретро — самые показательные из топ-5 скрипта):

| Застройщик | Медиана будни | Медиана выходные | Смысл |
|------------|---------------|------------------|-------|
| Ленстройтрест | ~100 мин | ~29 мин | быстрее на выходных |
| СКАТ | ~95 мин | ~57 мин | быстрее на выходных |
| Кронверк | ~2,5 сут | ~1,3 сут | медленно везде, но выходные чуть лучше |

Остальные из списка исключений в JSON: РГ-Девелопмент, ЭНКО — см. `retro-2026-q2-metrics.json` → `weekend_hypothesis.per_developer.exceptions_faster_on_weekend`.

**Оговорки.** Медианы по 3–6 заявкам на застройщика в срезе — **ориентир**, не статзначимость. У лидеров с медианой 1–2 мин в будни ratio «выходные / будни» раздувается (деление на почти ноль). Для решений по процессу полезнее смотреть и **долю молчащих**, и **phone book**, а не только этот срез.

**Дашборд.** Те же агрегаты по застройщику и рынку — KPI «Медиана скорости перезвона: Будни / Выходные» на вкладке «Обзор» (`first_call_by_day_type` в `data.json`; расчёт — [`shared/weekend-first-call.mjs`](../shared/weekend-first-call.mjs)).

## Часть IV. Рекомендации: что делать

### Приоритеты

| Когда | Действие | Эффект |
|-------|----------|--------|
| **До старта** | Phone book + верификация unknown-in-72h | +id. |
| **До старта** | Spam audit + dev-whitelist | +id., −ложный spam |
| **До старта** | Pre-flight 100 URL | −пропуски заявок |
| **Во время** | NTP, submit timestamp, notes | −«до заявки» |
| **Еженедельно** | top-30 unknown → phone book | +id. |
| **После цикла** | `export-retro-metrics`, `export-match-audit` | контроль KPI |

---

### A. Phone book (максимальный эффект на id.)

**Цель:** закрыть unknown, которые реально влияют на рейтинг.

```bash
cd projects/developer-response-rating
npm run export-phones-identify    # → data/working/phones_to_identify.xlsx
```

Сортировка в xlsx: сначала `priority_verify = высокий` (звонки в 72 ч на том же SIM), затем `call_count`.

**Верификация top-N (ручная):**

1. Перезвон / сайт застройщика / колл-центр.
2. Подтверждён → мастер `phone_book` + `developer_official_phones.xlsx`.
3. Только после проверки → `manual` в `phone_registry.json` (**модель DOGMA**).
4. **Не добавлять** без дозвона (**модель AVA**).

**Pre-collect до заявок** (100 застройщиков из `legend`):

- Телефоны с сайта, `tel:` в HTML, городские пулы (495, 812, 343…).
- SMS → call: для id. SMS (`sms_mark`) искать unknown с похожим префиксом.

**Применить:**

```bash
npm run seed-phones && npm run apply-phones
npm run export-retro-metrics   # проверить call_classification, q3_targets
```

**Еженедельно во время цикла:** повторить export → top-30 → мастер → pipeline.

---

### B. Калибровка spam

Широкие `DEFAULT_SPAM_RANGES` (`7963`, `7967`, …) дали **~715** auto-spam; часть может быть колл-центрами застройщиков.

**Правила:**

1. Подтверждённые spam-пулы — только **SPAM_BOOK** мастера → `npm run seed-spam`.
2. **Не расширять** spam-диапазоны «на глаз» в коде.
3. **Dev-whitelist выше spam** (пример: `796288` → AVA).
4. Не помечать spam номер, который звонил после заявки на том же SIM ≥3 раз у одного застройщика.

```bash
npm run audit-spam-ranges          # → spam-range-audit.xlsx
npm run suggest-spam-prefixes      # только для ручной проверки
```

| Ситуация в audit | Действие |
|------------------|----------|
| Номер застройщика подтверждён | PHONE_BOOK + dev-whitelist |
| Пул чисто спамовый | SPAM_PREFIXES в мастере |
| Сомнительно | unknown до верификации |

---

### D. Операционный протокол (снижение сирот «до заявки»)

Правило матча **не меняем**: `application_datetime ≤ event ≤ +72h`.

**Время на 21 устройстве:**

- NTP включён, часовой пояс **Europe/Moscow**, расхождение ≤ 1 мин перед волной.

**`application_datetime` = момент успешного submit**, не начало заполнения формы.

Порядок на заявку: открыть форму → отправить → **зафиксировать submit** → ждать входящие.

**Поле `notes` в applications** (спорные кейсы):

- URL формы после редиректов, скрин успеха, timestamp submit, ошибки/капча.

**Волны на SIM:** 21 SIM на ~100 застройщиков — норма; не смешивать два раунда в один день без паузы. После последней волны — **+3 суток** ожидания.

**Ответы после 72 ч:** не в KPI; видны в `company-events` при разборе с застройщиком.
